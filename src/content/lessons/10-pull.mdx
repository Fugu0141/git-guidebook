## なぜPullするのか

前のレッスンで、GitHub上のmainをMergeによって進めました。端末内のローカルmainは、Web上の操作だけでは更新されません。GitHubの最新履歴を取得し、ローカルmainへ反映します。

## 操作前の状態

GitHubのmainには学習目標がありますが、ローカルmainは1つ前のCommitを指しています。

## 手順：mainへ切り替えてPullする

1. GitHub Desktopの **Current Branch** から **main** を選びます。
2. 未Commit変更がないことを確認します。
3. 上部の **Fetch origin** を選び、GitHubの最新状態を確認します。
4. 表示が **Pull origin** に変わったら選びます。
5. 処理後、Changesが空で、履歴にMerge後のCommitがあることを確認します。
6. 作業フォルダのREADMEを開き、学習目標の行があることを確認します。

## 操作後の状態

作業フォルダのREADME、ローカルmain、GitHub上のmainが同じ変更を含みます。一連の操作で3つの場所が再びそろいました。

<Callout type="point" title="FetchとPull">
  Fetchはリモートに新しい履歴があるかを取得して確認します。Pullはその新しい履歴を現在のローカルBranchへ取り込み、通常は作業フォルダの内容も更新します。
</Callout>

## 内部で起きたこと

FetchでGitHub上のmainの位置を知り、Pullでローカルmainをその位置まで進めました。今回はローカルmainに独自の変更がなかったため、一直線に最新地点へ進みました。

<OptionalDetails summary="Pullの前に未Commit変更がある場合">
  変更内容によってはPullを安全に実行できません。慌てて変更を捨てず、どのBranchにいて何が未Commitか確認します。本格的な競合解決は次期コースで扱います。
</OptionalDetails>

## よくある失敗

作業BranchのままPullしてしまい、mainが更新されないことがあります。Current Branchをmainへ切り替えてから操作します。また、FetchだけでREADMEが変わるとは限りません。

## 対応するGitコマンド

作業フォルダでmainへ切り替え、未Commit変更がないことを確認します。Fetchは情報を取得し、Pullはリモートmainをローカルmainへ取り込みます。

```bash
git switch main
git fetch origin
git pull origin main
```

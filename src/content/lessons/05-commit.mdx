## なぜCommitするのか

編集した内容を「練習メモを追加した変更」というひとまとまりの履歴にします。後から見た人が目的を理解できるよう、Commitメッセージには結果ではなく変更の内容を書きます。

## 操作前の状態

READMEの変更は作業フォルダにあります。ローカルリポジトリにもGitHubにも、新しい履歴はまだありません。

## 手順：変更をCommitする

1. GitHub Desktopの **Changes** で `README.md` の差分をもう一度確認します。
2. 左下の **Summary (required)** に `Add practice note` と入力します。
3. **Description** は今回は空でも構いません。理由や補足が必要なときに使います。
4. ボタンが **Commit to main** になっていることを確認します。
5. **Commit to main** を選びます。
6. Changesが空になり、上部に **Push origin** が表示されることを確認します。

<Callout type="point" title="Commitはアップロードではありません">

</Callout>

## 操作後の状態

作業フォルダの内容とローカルリポジトリの最新Commitがそろいました。GitHubは1つ前の履歴のままです。

## 内部で起きたこと

選択した変更から新しいCommitが作られ、`main` のローカル履歴が1つ進みました。GitHub Desktopが「Push origin」を示すのは、まだGitHubへ送っていないCommitがあるからです。

<OptionalDetails summary="SummaryとDescription">
  Summaryは履歴一覧で最初に見える短い説明です。何を変えたかを命令形または簡潔な現在形で書きます。Descriptionには、なぜ必要か、注意点は何かなどを補足できます。
</OptionalDetails>

## よくある失敗

`update` や `fix` だけでは変更内容が伝わりません。`Add practice note` のように対象と操作が分かる文にします。また、秘密情報が差分にないか、Commit前に必ず読みます。

## 対応するGitコマンド

作業フォルダで、変更を選んでからCommitします。`git add` は次のCommitに含める変更を選び、`git commit` はローカルリポジトリへ履歴を作ります。

```bash
git add README.md
git commit -m "Add practice note"
```

## なぜ履歴が必要なのか

文章や資料を直すたびに、`完成版.docx`、`完成版_修正.docx`、`完成版_本当の最終.docx`とコピーした経験はありませんか。コピーはその瞬間の内容を残せますが、「どれが最新か」「何を変えたか」「なぜ変えたか」が分かりにくくなります。

Gitは、ファイルを増殖させる代わりに、ひとつの作業場所で変更のまとまりを履歴として記録します。必要なときは、過去の状態や変更内容を確認できます。

<Callout type="point" title="Gitが記録するもの">
  Gitは「完成したファイル」だけでなく、どのファイルのどこを変え、その変更をどんな意図で記録したかを扱います。
</Callout>

## このコースで作るもの

これからGitHubに `git-guidebook-practice` という練習用リポジトリを作り、端末へCloneします。自己紹介ファイルを編集してCommit、Pushし、Branch上の変更をPull RequestでmainへMergeします。最後に、GitHub上で進んだmainをPullして端末へ取り込みます。

手順は次の順です。

1. GitHub上にリモートリポジトリを作る
2. Cloneして、端末に作業フォルダとローカルリポジトリを作る
3. ファイルを編集し、差分を確認する
4. Commitでローカルの履歴へ記録する
5. Pushでその履歴をGitHubへ送る
6. Branchを作り、Pull RequestをMergeする
7. PullでGitHubの最新履歴を端末へ取り込む

## 内部で起きたこと

まだ何も作成・変更していません。「どこが変わったか」を確かめる考え方を準備した段階です。

<OptionalDetails>
  Gitはスナップショットという考え方で各Commitの状態を記録します。初心者のうちは、Commitを「意味のある変更をひとまとまりにした履歴」と考えれば十分です。
</OptionalDetails>

## よくある失敗

「Gitを使えば自動で全部保存される」と思い込むことです。作業中のファイル保存と、Gitで履歴を記録するCommitは別の操作です。

## 対応するGitコマンド

このレッスンでは履歴を変えるコマンドを実行しません。Gitを使える環境か確認したい場合だけ、任意のフォルダで次を実行できます。バージョンを表示するだけで、ファイルや履歴は変わりません。

```bash
git --version
```

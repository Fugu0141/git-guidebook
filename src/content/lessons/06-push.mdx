## なぜPushするのか

Commitした履歴は、まだ自分の端末内にしかありません。GitHub上でも確認・共有できるように、ローカルのCommitをリモートリポジトリへ送ります。この操作がPushです。

## 操作前の状態

ローカルのmainはGitHub上のmainよりCommitが1つ進んでいます。作業フォルダに未Commit変更はありません。

## 手順：Pushして確認する

1. GitHub Desktop上部の **Push origin** を選びます。
2. 処理が終わり、ボタンが **Fetch origin** に変わることを確認します。
3. **Repository → View on GitHub** を選びます。
4. GitHub上のREADMEに「練習メモ」があり、最新Commitが `Add practice note` になっていることを確認します。

## 操作後の状態

ローカルとGitHub上のmainが同じCommitを指しています。Pushは作業フォルダを編集する操作ではないため、端末内のファイル内容は変わりません。

<Callout type="point" title="CommitとPushの違い">
  Commitは選んだ変更をローカルの履歴へ記録します。PushはローカルにあるCommit履歴をリモートへ送ります。Pushは「いま開いているファイルだけ」を送る操作ではありません。
</Callout>

## 内部で起きたこと

Gitは `origin` と呼ばれるリモートへ、ローカルのmainにある新しいCommitを送信しました。GitHub側のmainが同じCommitまで進みました。

<OptionalDetails summary="Fetch originと表示される理由">
  Push後は送るCommitがありません。GitHub
  Desktopは、リモートに新しい履歴がないか確認するFetchを次の候補として表示します。Fetchは情報を取得しますが、作業フォルダへ内容を反映するPullとは異なります。
</OptionalDetails>

## よくある失敗

ネットワークやサインインの問題でPushが完了しないことがあります。何度もボタンを押す前に、表示されたメッセージ、接続、GitHubアカウントを確認してください。エラーに秘密情報が含まれる場合は公開場所へ貼らないでください。

## 対応するGitコマンド

作業フォルダで、ローカルmainに未送信Commitがある状態で実行します。成功するとリモートのmainが進みます。

```bash
git push origin main
```

## 次のレッスンへ

次はmainを直接進めず、新しいBranchへ履歴を分けて安全に変更します。

## なぜMergeするのか

Pull Requestは提案を作る段階でした。差分が正しいと確認できたら、Mergeで作業Branchの変更をmainの履歴へ統合します。

## 操作前の状態

GitHub上では作業Branchがmainより進み、Pull RequestはOpenです。ローカルのmainはまだ変更前です。

## 手順：確認してMergeする

1. Pull Requestの **Files changed** で、READMEの1行追加だけが含まれることを再確認します。
2. 会話画面に戻り、競合がなくMerge可能と表示されていることを確認します。
3. **Merge pull request** を選びます。
4. タイトルを確認し、**Confirm merge** を選びます。
5. Pull Requestが **Merged** になったことを確認します。
6. **Delete branch** は、今後この作業Branchを使わない場合に選びます。

<Callout type="warning" title="Merge前に止まって確認">
  baseがmainであること、意図したファイルだけが変わること、秘密情報がないことを確認します。分からない差分があればMergeしません。
</Callout>

## 操作後の状態

GitHub上のmainが作業Branchの変更を含む地点まで進みました。ただし、端末内のローカルmainは自動では進みません。この差を次のPullで解消します。

## 内部で起きたこと

GitHub上でmainの履歴が更新され、Pull RequestはMergedとして閉じました。作業Branchを削除しても、mainへ統合されたCommitや変更は消えません。削除されるのは不要になったBranch名です。

<OptionalDetails summary="Merge方式について">
  GitHubにはMerge commit、Squash and merge、Rebase and
  mergeなどがあります。この教材ではリポジトリの既定のMerge方法を使います。違いの詳しい説明はMVPの範囲外です。
</OptionalDetails>

## よくある失敗

Mergeが終わると端末内も更新されたと思い込むことがあります。Web上の操作はGitHub側で起きています。ローカルmainへ切り替え、Pullするまで端末側のmainは古いままです。

## 対応するGitコマンド

このレッスンではGitHub上のPull RequestをMergeしました。同じリポジトリ内での基本的なMergeは、取り込み先Branchへ切り替えて実行しますが、今回は参考だけにします。

```bash
git switch main
git merge add-learning-goal
```

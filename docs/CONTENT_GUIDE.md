# 教材執筆ガイド

## 1. 基本方針

この教材では、初心者に正しい用語を一度に覚えさせるのではなく、操作の目的と流れを先に理解してもらいます。

ただし、分かりやすさのために技術的に誤った説明をしてはいけません。最初は簡単に説明し、必要に応じて「もう少し正確に」の欄で補足します。

## 2. 1レッスンの標準構成

1. 今回できるようになること
2. なぜこの操作が必要か
3. 操作前の状態
4. 手順
5. 操作後の状態
6. 内部で起きたこと
7. よくある失敗
8. 確認問題
9. 対応するGitコマンド
10. 次のレッスンとのつながり

## 3. 文体

- です・ます調を基本にする
- 一文を長くしすぎない
- 初出の用語には日本語で意味を添える
- 操作名はGitHub Desktop上の表記と一致させる
- 「押せばOK」で終わらず、押した結果を説明する
- 不必要に怖がらせない
- 初心者の疑問を責める表現を使わない

## 4. 用語の説明

用語は次の順で説明します。

1. 何のために必要か
2. 日常語に近い説明
3. 実際の操作例
4. 技術的により正確な説明
5. 他の用語との違い

例：Commit

> まず理解する説明  
> Commitは、現在の変更をひとまとまりの履歴として自分の端末内に記録する操作です。

> もう少し正確に  
> GitHub Desktopで選択した変更をもとに、Gitリポジトリへ新しいCommitを作成します。この時点ではGitHub上のリポジトリには送信されていません。

## 5. 状態を明示する

各操作では、最低限次の3か所を区別します。

- 作業フォルダ
- ローカルリポジトリ
- GitHub上のリモートリポジトリ

説明中は「保存された」「反映された」だけで終わらず、どこに反映されたのかを書きます。

悪い例：

> Commitすると変更が保存されます。

良い例：

> Commitすると、選択した変更がローカルリポジトリの履歴として記録されます。まだGitHub上には送られていません。

## 6. スクリーンショット

- 画像だけに重要な情報を持たせない
- クリック箇所を文章でも示す
- 個人情報、アクセストークン、メールアドレスを映さない
- UI更新で古くなった画像には対象バージョンを添える
- 可能な限り同じ画面サイズとテーマで撮影する
- 画像には代替テキストを付ける

## 7. 図解

図は装飾ではなく、現在地を理解するために使います。

- 操作前と操作後を比較できるようにする
- データや履歴が動く方向を矢印で示す
- 色だけで状態を区別しない
- Branchは単なるフォルダのコピーとして描かない
- Commit、Branch、HEADなどは説明レベルに応じて簡略化する

## 8. コマンドの扱い

MVPではGitHub Desktopでの理解を優先します。各レッスン末尾で、対応するコマンドを参考として示します。

コマンドは次の情報とセットにします。

- どのフォルダで実行するか
- 実行前提
- 実行すると何が変わるか
- GitHub Desktopのどの操作に近いか

## 9. 確認問題

確認問題は用語暗記だけにしません。

推奨する形式：

- 現在の状態から次に必要な操作を選ぶ
- CommitとPushのどちらが必要か判断する
- 操作後にどこが更新されるか答える
- 間違った操作例の問題点を説明する

## 10. レビュー時の確認項目

- 説明の対象読者が明確か
- 操作する理由が先に説明されているか
- ローカルとリモートが混同されていないか
- CommitとPushが混同されていないか
- Pull RequestとMergeが混同されていないか
- 手順が現在のUIと一致しているか
- 画像なしでも最低限理解できるか
- キーボードやスクリーンリーダー利用者を排除していないか
- 次のレッスンへ自然につながるか
